# AI·데이터 기반 콘텐츠 전략가를 위한 RSS 트렌드 브리핑 아카이브

## 1. 프로젝트 제목

**AI·데이터 기반 콘텐츠 전략가를 위한 RSS 트렌드 브리핑 아카이브**

> RSS 피드와 공식 업데이트 페이지를 자동 수집하여, AI·데이터 기반 콘텐츠 기획자가 매일 확인해야 할 마케팅, 플랫폼, AI, 데이터 트렌드를 한 곳에서 확인할 수 있도록 만든 서버리스 브리핑 시스템입니다.

---

## 2. 프로젝트 취지

콘텐츠 기획자는 단순히 감각이나 유행에만 의존해서는 안 된다.
AI, 데이터, 플랫폼, 소비자 트렌드가 빠르게 변하는 환경에서는 여러 공식 채널과 뉴스 소스를 지속적으로 모니터링하고, 그중 실제 기획에 활용할 만한 정보를 선별하는 능력이 중요하다.

이 프로젝트는 여러 사이트를 직접 방문하지 않고도 RSS 피드와 공식 업데이트 페이지를 자동으로 수집하고, 수집된 글을 AI가 중요도 기준에 따라 평가하여 일일 브리핑 형태로 정리하는 것을 목표로 한다.

단, 이 프로젝트는 모든 피드를 30분마다 수집하지 않고, 뉴스성 RSS와 기술블로그·AI 연구 피드를 분리하여 운영한다.

또한 별도 서버 없이 **GitHub Actions + GitHub Pages**만 사용하여 유지비용 0원으로 운영할 수 있도록 설계한다.

### 핵심 목표

* RSS 피드 자동 수집
* 뉴스성 피드와 기술/AI 피드의 수집 주기 분리
* 새 글 감지 및 중복 제거
* AI 기반 중요도 평가 및 요약
* 일자별 Markdown 브리핑 아카이브 생성
* GitHub Pages 기반 정적 대시보드 제공
* 당일 저녁 또는 다음날 아침 이메일 브리핑 발송
* 무료 인프라만 사용한 포트폴리오 프로젝트 구현

## 3. 주요 기능

### 3.1 RSS 피드 자동 수집

RSS/Atom을 제공하는 사이트는 `feedparser`를 사용해 수집한다.

수집 항목은 다음과 같다.

* 제목
* 요약
* 링크
* 발행일
* 출처
* 카테고리
* 수집 시각
* 고유 ID

---

### 3.2 수집 주기 분리

모든 RSS를 같은 주기로 수집하지 않는다.
피드 성격에 따라 `group`을 나누고, 각 group별 GitHub Actions workflow를 분리한다.

| 피드 유형                        | group             | 수집 주기    | 이유                                       |
| ---------------------------- | ----------------- | -------- | ---------------------------------------- |
| 뉴스성 RSS                      | `news`            | 30분마다    | 경제·비즈니스·기술 뉴스는 업데이트가 잦고 트렌드 감지 가치가 높음    |
| 국내 플랫폼·서비스 기술블로그             | `tech_blog`       | 6시간마다    | 업데이트 빈도는 낮지만 제품·UX·데이터 실무 사례 파악에 중요      |
| AI 연구·AI 기업 블로그              | `ai_data`         | 6시간~1일마다 | 논문·모델·오픈소스 업데이트는 실시간보다 선별이 중요            |
| 공식 changelog / release notes | `official_update` | 하루 1~2회  | API·제품 변경사항은 정확한 감지가 중요하지만 초단위 모니터링은 불필요 |
| 일일 브리핑 생성                    | `briefing`        | 하루 1회    | 수집된 항목을 선별·요약하여 아카이브화                    |

---

### 3.3 새 글 감지

수집된 글의 `title + url`을 기준으로 고유 ID를 생성한다.
이전 실행 결과와 비교하여 새 글만 감지한다.

새 글이 없으면 다음 작업을 수행하지 않는다.

* `docs/data.json` 갱신하지 않음
* 이메일 발송하지 않음
* Markdown 아카이브 갱신하지 않음
* Git commit 하지 않음

이 로직을 통해 30분마다 workflow가 실행되더라도 저장소 커밋 로그가 불필요하게 쌓이지 않도록 한다.

---

### 3.4 AI 기반 중요도 평가

수집된 항목 중 콘텐츠 기획에 활용할 만한 글을 선별하기 위해 AI를 활용한다.

평가 기준은 다음과 같다.

* 콘텐츠 기획에 활용 가능한가?
* 소비자, 플랫폼, 기술 트렌드와 관련 있는가?
* AI 또는 데이터 기반 의사결정에 도움이 되는가?
* 실제 콘텐츠 아이디어나 서비스 기획으로 확장 가능한가?
* 단순 사건이 아니라 반복적 흐름이나 변화의 신호를 보여주는가?
* 여러 소스에서 반복적으로 등장하는 키워드와 연결되는가?

각 글에는 중요도 점수를 부여한다.

```json
{
  "importance_score": 87,
  "keywords": ["AI 마케팅", "숏폼", "개인화"],
  "reason": "AI 기반 광고 자동화와 콘텐츠 개인화 흐름을 보여주는 업데이트",
  "planning_insight": "브랜드 콘텐츠 기획 시 AI 추천과 개인화 전략을 함께 고려할 필요가 있음"
}
```

---

### 3.5 일일 브리핑 아카이브 생성

매일 수집된 주요 글을 Markdown 파일로 저장한다.

```text
docs/archive/
  2026-06-24.md
  2026-06-25.md
```

브리핑에는 다음 내용이 포함된다.

* 오늘의 핵심 요약
* 중요도 높은 Top 10 이슈
* 반복 등장한 키워드
* 콘텐츠 기획 인사이트
* 원문 링크

---

### 3.6 이메일 브리핑

새 글이 올라올 때마다 즉시 이메일을 보내지 않는다.
이 프로젝트는 알림 시스템이 아니라 브리핑 시스템이므로, 중요도 기준을 통과한 항목을 모아 당일 저녁 또는 다음날 아침 이메일로 발송한다.

알림 기준 예시:

* 중요도 점수 70점 이상
* 핵심 키워드 포함
* 공식 changelog 또는 platform update에 해당
* 콘텐츠 기획 또는 서비스 전략으로 확장 가능한 이슈

---

### 3.7 GitHub Pages 대시보드

수집 결과는 `docs/data.json`에 저장한다.
GitHub Pages는 이 JSON 파일을 불러와 웹페이지에 최신 브리핑을 표시한다.

`docs/data.json`은 웹페이지 표시용이므로 최신 100~200개 항목만 유지한다.
장기 기록은 `docs/archive/*.md`에 일자별로 저장한다.

### 4.2 개발 단계

1차 MVP에서는 RSS 수집 기능을 우선 구현한다.
스크래핑 기능은 처음부터 넣지 않고, 공식 changelog와 동적 페이지 모니터링이 필요해지는 2차 이후 단계에서 추가한다.

| 단계   | 범위                                                              |
| ---- | --------------------------------------------------------------- |
| v1   | RSS 19개 수집, 새 글 감지, `docs/data.json` 저장, GitHub Pages 표시        |
| v1.5 | AI 중요도 평가, Top 10 브리핑 생성, Markdown 아카이브 저장                      |
| v2   | OpenAI, Gemini, Google Ads 등 공식 changelog BeautifulSoup 스크래핑 추가 |
| v3   | TikTok, Meta, Instagram 등 Playwright 기반 동적 페이지 스크래핑 추가          |

---

### 4.3 작동 흐름

```text
1. GitHub Actions 정기 실행

   ├── 뉴스성 RSS
   │   └── 30분마다 수집
   │
   ├── 기술블로그 RSS
   │   └── 6시간마다 수집
   │
   └── AI·데이터 RSS
       └── 6시간~1일 단위 수집

        ↓

2. scripts/scraper.py 실행

   ├── config.json 읽기
   ├── group에 맞는 피드만 선택
   ├── RSS 피드 수집
   └── feedparser로 파싱

        ↓

3. 새 글 여부 확인

        ↓

4. 새 글이 있으면 docs/data.json 업데이트

        ↓

5. AI 중요도 평가 및 요약

        ↓

6. docs/archive/YYYY-MM-DD.md 생성 또는 갱신

        ↓

7. 중요 글이 있으면 이메일 브리핑 발송

        ↓

8. 변경사항이 있을 때만 GitHub Actions bot이 자동 commit

        ↓

9. GitHub Pages에서 최신 브리핑 표시
```

## 5. config.json 설계

이 프로젝트의 핵심 설정은 `config.json`에서 관리한다.

단순히 피드 이름과 URL만 저장하지 않고, 수집 주기 분리와 AI 중요도 평가를 위해 다음 필드를 함께 저장한다.

| 필드                  | 설명                                                       |
| ------------------- | -------------------------------------------------------- |
| `name`              | 피드 이름                                                    |
| `type`              | 수집 방식: `rss`, `scrape`, `playwright`                     |
| `url`               | RSS 또는 수집 대상 URL                                         |
| `group`             | 수집 그룹: `news`, `tech_blog`, `ai_data`, `official_update` |
| `category`          | 세부 분류                                                    |
| `icon`              | 웹페이지 표시용 아이콘                                             |
| `enabled`           | 활성화 여부                                                   |
| `importance_weight` | 출처 신뢰도 또는 중요도 가중치                                        |
| `language`          | `ko` 또는 `en`                                             |
| `max_items`         | 피드별 최대 수집 개수                                             |

예시:

```json
{
  "feeds": [
    {
      "name": "ZDNet Korea",
      "type": "rss",
      "url": "https://feeds.feedburner.com/zdkorea",
      "group": "news",
      "category": "tech_news",
      "icon": "💻",
      "enabled": true,
      "importance_weight": 0.85,
      "language": "ko",
      "max_items": 20
    },
    {
      "name": "토스 기술블로그",
      "type": "rss",
      "url": "https://toss.tech/rss.xml",
      "group": "tech_blog",
      "category": "product_tech",
      "icon": "💳",
      "enabled": true,
      "importance_weight": 0.9,
      "language": "ko",
      "max_items": 10
    },
    {
      "name": "Hugging Face Blog",
      "type": "rss",
      "url": "https://huggingface.co/blog/feed.xml",
      "group": "ai_data",
      "category": "open_source_ai",
      "icon": "🤗",
      "enabled": true,
      "importance_weight": 0.9,
      "language": "en",
      "max_items": 10
    }
  ]
}
```

이 구조를 사용하면 다음 작업이 쉬워진다.

* 피드별 활성화/비활성화
* group별 수집 주기 분리
* category별 필터링
* 출처별 중요도 가중치 반영
* 웹페이지에서 피드별 보기 제공
* AI 점수화 시 출처 가중치 적용

## 6. 1차 MVP RSS 피드 구성

1차 MVP에서는 수집 대상을 과도하게 늘리지 않고, 콘텐츠 전략가에게 직접적으로 도움이 되는 핵심 RSS 19개로 시작한다.

---

### 6.1 뉴스성 RSS: 30분마다 수집

뉴스성 피드는 경제·비즈니스·기술 변화가 빠르게 반영되므로 30분마다 수집한다.

| 출처             | RSS URL                                                       | group  | 활용 목적             |
| -------------- | ------------------------------------------------------------- | ------ | ----------------- |
| 한국경제           | `https://rss.hankyung.com/economy.xml`                        | `news` | 국내 경제·산업 트렌드      |
| 매일경제 기업        | `https://www.mk.co.kr/rss/50100032/`                          | `news` | 기업 전략, 산업 변화      |
| 머니투데이          | `https://news.mt.co.kr/rss/news_total.xml`                    | `news` | 스타트업, 커머스, 소비 트렌드 |
| ZDNet Korea    | `https://feeds.feedburner.com/zdkorea`                        | `news` | 테크 제품, 플랫폼 변화     |
| BBC Technology | `https://feeds.bbci.co.uk/news/technology/rss.xml`            | `news` | 글로벌 기술 변화         |

### 6.3 AI·데이터 트렌드 RSS: 6시간~1일 단위 수집

AI·데이터 관련 피드는 실시간 감시보다 의미 있는 업데이트를 선별하는 것이 중요하다.

| 출처           | RSS URL                                       | group     | 활용 목적               |
| ------------ | --------------------------------------------- | --------- | ------------------- |
| arXiv cs.CL  | `https://rss.arxiv.org/rss/cs.CL`             | `ai_data` | NLP, LLM 연구 동향      |

`docs/data.json`은 GitHub Pages 웹페이지 표시용 데이터이다.

## 9. GitHub Actions 운영 방식

이 프로젝트는 별도 서버를 사용하지 않는다.
GitHub Actions의 scheduled workflow를 활용해 정해진 시간마다 자동 실행한다.

사용자의 PC가 꺼져 있어도 GitHub-hosted runner에서 `scripts/scraper.py`가 실행된다.
수집 결과는 `docs/data.json`과 `docs/archive/*.md`에 저장되고, GitHub Pages에서 정적 웹페이지로 표시된다.

---

### 9.1 뉴스성 RSS workflow 예시

뉴스성 RSS는 30분마다 실행한다.

```yaml
name: News RSS Monitor

on:
  schedule:
    - cron: "*/30 * * * *"
  workflow_dispatch:

permissions:
  contents: write

jobs:
  scrape-news:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run news RSS scraper
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          EMAIL_USER: ${{ secrets.EMAIL_USER }}
          EMAIL_PASSWORD: ${{ secrets.EMAIL_PASSWORD }}
          EMAIL_TO: ${{ secrets.EMAIL_TO }}
        run: python scripts/scraper.py --group news

      - name: Commit updated data
        run: |
          git config user.name "github-actions"
          git config user.email "github-actions@github.com"
          git add docs/data.json docs/archive || true
          git diff --cached --quiet || git commit -m "Update news RSS data"
          git push
```

---

## 10. 보안 및 운영 원칙

### 10.1 Secrets 관리

다음 값은 코드나 `config.json`에 직접 작성하지 않는다.
반드시 GitHub Secrets에 저장한다.

```text
GEMINI_API_KEY
OPENAI_API_KEY
EMAIL_USER
EMAIL_PASSWORD
EMAIL_TO
```

### 10.2 로그 관리

GitHub Actions 로그에 API 키나 이메일 비밀번호가 출력되지 않도록 한다.

금지 예시:

```python
print(os.environ["GEMINI_API_KEY"])
print(email_password)
```

허용 예시:

```text
✅ ZDNet Korea: 12건 수집
✅ 신규 글 3건 감지
✅ 브리핑 생성 완료
```

### 10.3 workflow 권한 최소화

기본 MVP에서는 저장소에 커밋할 권한만 필요하다.

```yaml
permissions:
  contents: write
```

`pages: write`, `id-token: write` 등은 GitHub Actions로 Pages를 직접 배포하는 구조에서만 사용한다.

### 10.4 외부 PR 보안

공개 저장소로 운영할 경우 외부 PR에서 Secrets가 실행되지 않도록 한다.
초기 workflow trigger는 다음 두 가지만 사용한다.

```yaml
on:
  schedule:
  workflow_dispatch:
```

`pull_request` 자동 실행은 초기에 사용하지 않는다.
